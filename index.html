Que significa cada dato

DatoQue mideComo interpretarlo
Cobertura global (93.8%) Porcentaje de lineas de codigo que fueron ejecutadas al menos una vez durante la suite de tests. Un valor alto indica que los tests recorren la mayoria del codigo. Por debajo del 80% suele considerarse insuficiente para proyectos en produccion.
Metodos cubiertos (88.5%) Proporcion de metodos/funciones que fueron invocados por algun test. Complementa la cobertura de lineas: un metodo puede estar cubierto en lineas pero nunca llamado directamente. Valores altos aqui indican una suite robusta.
Lineas cubiertas por paquete Cobertura de lineas desagregada por capa de la aplicacion (controller, service, model, etc.). Permite identificar que capas estan bien testeadas y cuales necesitan atencion. Verde ≥ 80%, amarillo 50–79%.
Lineas cubiertas por clase Cobertura individual de cada clase dentro del proyecto. Util para detectar clases especificas con baja cobertura. Una clase en amarillo no necesariamente es un problema si sus caminos no cubiertos son poco probables en produccion.
Suite de tests ✓ Estado de ejecucion de todos los tests al momento de generar el reporte. Si algun test falla el build no pasa y el reporte no se genera. El check verde confirma que todos los tests pasaron correctamente.
Paquetes excluidos config, dto y repository fueron removidos del calculo. config no tiene logica testeable. dto tiene builders generados por Lombok que inflarian el denominador. repository son interfaces del framework, no codigo propio.

Vision general de calidad

Que significa cada dato

✅
Quality Gate: Aprobado

SonarCloud define umbrales minimos de calidad. Si el codigo los supera, el gate esta en verde. Esto significa que el proyecto cumple los estandares de calidad acordados para poder desplegarse a produccion.

🐛
Bugs (0)

Errores de logica que SonarCloud detecta de forma estatica, sin ejecutar el codigo. Tener 0 bugs es el resultado ideal: significa que no hay patrones de comportamiento incorrecto identificables automaticamente.

🔐
Vulnerabilidades (4) — Requieren atencion

Son puntos del codigo donde un atacante podria explotar una debilidad: manejo inseguro de datos, configuraciones expuestas, validaciones faltantes, etc. Con 4 vulnerabilidades la calificacion de seguridad es E, la mas baja. El equipo tiene identificadas estas vulnerabilidades y estan planificadas para el proximo sprint de seguridad.

🔍
Hotspot de Seguridad (1, 50% revisado)

A diferencia de las vulnerabilidades, un hotspot es una zona que PODRIA ser riesgosa pero requiere revision humana para determinarlo. El equipo ya reviso el 50% del unico hotspot existente.

🧹
Code Smells (10) y Deuda Tecnica (64 min)

Los code smells son practicas de escritura que no causan errores pero dificultan el mantenimiento futuro. La deuda tecnica es el tiempo estimado para corregirlos: 64 minutos sobre 2115 lineas de codigo es un resultado excelente, indica un codigo limpio y bien estructurado.

📋
Cobertura (91%) y Duplicacion (2.1%)

SonarCloud confirma el 91% de cobertura de tests, alineado con el reporte JaCoCo. La duplicacion del 2.1% esta muy por debajo del umbral tipico del 3%, lo que indica que el equipo evita copiar y pegar codigo.
